01
Preference architecture

One toggle list, three surfaces

Which apps show in the nav — and in what order — is a single JSON column on the user's profile row. The same reorder editor renders inside desktop settings and the mobile drawer's settings sheet, and both write to the same preference, which also drives the Home page's shortcut tiles. Change the order once, it's consistent everywhere, immediately.

function resolvedNavApps() { // falls back to every module, in a fixed default order, // until the user has actually saved a preference if (!navAppsPref) return APP_MODULES.map(m => ({ ...m })); return navAppsPref .filter(p => p.visible) .map(p => APP_MODULES.find(m => m.app === p.app)) .filter(Boolean); }

03
Data integrity

A "best" that can't lie

Logging a max-effort attempt doesn't make it a personal record — only a confirmed success does. The weight is withheld from every "best" calculation (the continuity card, the session comparison, the dashboard's PR feed) until the lift is marked successful; a failed or still-unconfirmed attempt is recorded — and visibly tagged — without ever being able to inflate the number that everything else is measured against.

.or("is_pr_attempt.eq.false,pr_result.eq.success") // excludes: failed attempts, and attempts still awaiting confirmation
04
Component from scratch

A <select> that doesn't clip

The native dropdown looks like every browser's default and clips long option text. The custom select renders its own popover against a hidden native <select>, so keyboard and form-submission behavior stay standard while the visible chrome is fully the app's own — no accessibility trade-off for the redesign.

01
Gesture coordination

Touches don't fight each other

A phone screen has one touch surface and three gestures that all want it: the edge-swipe drawer, swipe-to-reveal-actions on a row, and drag-to-reorder. An "owns-gesture" marker lets a row opt its own touch stream out of the drawer's listener before the two ever collide — no gesture silently overrides another. 02
Resilience

An outage doesn't erase your data

A failed write used to mean the entry was just gone. Every logging path now generates its own row ID client-side before the network call, so a connectivity failure queues the write locally (localStorage) instead of losing it — the UI proceeds optimistically, a small badge shows what's still syncing, and the client-generated ID makes the eventual retry an idempotent upsert, not a risky duplicate insert.

const localId = crypto.randomUUID(); // insert now proceeds locally even if this call ends up queued — // the retry later replays onto the same row, safely, via upsert writeWithQueue("workout_sets", "upsert", { id: localId, ... });
03
Motion

FLIP, not a jump-cut

Reordering a list by dragging is easy to get janky: measure the First position, Last position, Invert the difference with a transform, then let it Play to zero — the displaced rows glide into their new slots instead of teleporting, and the real DOM reorder only happens once, on drop.

04
Input disambiguation

A tap isn't a scroll, until it moves

Binding a pressed-state animation to touchstart means every button your thumb glides over mid-scroll flashes as if it were tapped. Movement direction decides which gesture wins — vertical drift clears the pressed state (it was a scroll), while horizontal drift is left alone, because that's a swipe gesture's job to interpret, not this one's.
